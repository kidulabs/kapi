# Project Structure

## Overview

A kapi controller project created by `kapibuild init` follows a standard layout
designed to keep API types, controller logic, and schemas organized as the
project grows.

## Directory Layout

```
my-controller/
├── Cargo.toml              # Project manifest
├── Kapifile                # Project metadata (YAML)
├── api/                    # API type definitions
│   └── example.io/         # Group directory
│       ├── mod.rs          # Re-exports all versions
│       ├── v1/             # Version directory
│       │   ├── mod.rs      # Re-exports all kinds
│       │   ├── widget.rs   # Widget spec and status structs
│       │   └── gadget.rs   # Gadget spec and status structs
│       └── v2/             # (future) Next version
│           └── widget.rs
├── schemas/                # Generated JSON Schema files
│   └── example.io_Widget.json
└── src/
    ├── main.rs             # Entry point with Manager setup
    └── controllers/        # Controller implementations
        ├── mod.rs          # Module declarations
        └── widget_controller.rs
```

## File Purposes

### `Cargo.toml`

The project manifest with dependencies on the kapi workspace crates:

```toml
[package]
name = "my-controller"
version = "0.1.0"
edition = "2021"

[dependencies]
kapi-controller = { path = "../kapi-controller" }
kapi-client = { path = "../kapi-client" }
kapi-core = { path = "../kapi-core" }
kapi-derive = { path = "../kapi-derive" }
serde = { version = "1", features = ["derive"] }
serde_json = "1"
tokio = { version = "1", features = ["full"] }
tracing = "0.1"
tracing-subscriber = "0.3"
async-trait = "0.1"
schemars = "0.8"
```

### `Kapifile`

YAML metadata file tracking project resources:

```yaml
domain: example.io
resources:
  - kind: Widget
    version: v1
    scope: Namespaced
    has_status: true
  - kind: Gadget
    version: v1
    scope: Namespaced
    has_status: false
```

The `resources` list is automatically updated by `kapibuild create api` and is
used by future kapibuild features.

### `api/` Directory

Contains the Rust type definitions for your API resources. Organized by group
and version following this pattern:

```
api/{group}/{version}/{kind}.rs
```

Each `{kind}.rs` file defines the spec and (optionally) status structs for that
resource kind with the `#[derive(KapiResource)]` macro:

```rust
// api/example.io/v1/widget.rs
use kapi_derive::KapiResource;
use schemars::JsonSchema;
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, KapiResource, Serialize, Deserialize, JsonSchema)]
#[kapi(group = "example.io", version = "v1", kind = "Widget")]
pub struct WidgetSpec {
    pub color: String,
    pub replicas: u32,
}
```

### `schemas/` Directory

Contains generated JSON Schema files, one per resource:

```
schemas/{group}_{kind}.json
```

These files are generated by `kapibuild generate` and are ready to be registered
with the kapi server via the Schema API.

### `src/main.rs`

Application entry point that sets up tracing, creates a kapi client, initializes
the Manager, wires controllers, and starts the event loop:

```rust
#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error + Send + Sync>> {
    tracing_subscriber::fmt::init();
    let client = KapiClient::new("http://localhost:8080")?;
    let mut manager = Manager::new(client);
    manager
        .controller_for(Widget::key())
        .reconcile_with(WidgetReconciler)
        .register();
    manager.start().await?;
    Ok(())
}
```

### `src/controllers/` Directory

Contains one file per controller. Each file exports a reconciler struct that
implements the `Reconciler` trait.

## Module Organization

The Rust module tree mirrors the directory structure:

```
crate
├── api
│   └── example.io
│       └── v1
│           ├── widget (WidgetSpec, WidgetStatus)
│           └── gadget (GadgetSpec)
├── controllers
│   └── widget_controller (WidgetReconciler)
└── main
```

The version `mod.rs` files re-export all kinds for convenient importing:

```rust
// In src/main.rs or a controller:
use api::example::io::v1::WidgetSpec;
use api::example::io::v1::GadgetSpec;
```

## Multiple Resources Per Project

You can add multiple API resources to the same project. Each gets its own
`{kind}.rs` file in the version directory:

```
api/example.io/v1/
├── mod.rs          # Re-exports both widget and gadget
├── widget.rs       # WidgetSpec, WidgetStatus
└── gadget.rs       # GadgetSpec
```

Adding resources in different groups or versions is also supported.
