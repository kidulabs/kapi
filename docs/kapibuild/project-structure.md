# Project Structure

The `kapibuild init` command scaffolds a new kapi controller project with the following directory layout:

```
<project-name>/
├── Cargo.toml          # Project manifest with kapi dependencies
├── Kapifile            # YAML manifest tracking resources and domain
├── schemas/            # JSON schemas (generated by `kapibuild api generate`)
└── src/
    ├── main.rs         # Entry point with Manager setup
    ├── api/             # API type definitions (by `kapibuild api create`)
    ├── types/           # Generated typed wrappers (by `kapibuild api generate`)
    └── controllers/    # Controller implementations (by `kapibuild controller generate`)
        └── mod.rs
```

## File Descriptions

### Cargo.toml

Contains project metadata and dependencies:
- **kapi-core**: Core types (StoredObject, WatchEvent, ResourceKey)
- **kapi-client**: HTTP client for communicating with kapi servers
- **kapi-controller**: Controller runtime SDK (Manager, Reconciler traits)
- **serde**: Serialization/deserialization with derive support
- **tokio**: Async runtime
- **tracing**: Structured logging
- **async-trait**: Async trait support
- **schemars**: JSON Schema generation from Rust types

All kapi dependencies use version numbers matching the kapibuild tool version. The generated project is standalone and can be built independently of the kapi workspace.

### Kapifile

YAML manifest tracking project metadata:
```yaml
domain: kapi.io
version: v1
resources: []
```

- **domain**: API group for resources (e.g., `kapi.io`)
- **version**: Default API version (e.g., `v1`)
- **resources**: List of resources managed by this project

### src/api/

Directory for API type definitions. Use `kapibuild api create` to generate types here.

### src/types/

Directory for generated typed wrapper structs. The `kapibuild api generate` command creates wrapper files here:

- `src/types/<group_mod>/<version>/<kind>.rs` — Wrapper struct with `metadata`, `system`, `spec`, and optional `status` fields
- Implements `key()`, `schema_data()`, and the `TypedResource` trait for use with `TypedClient`
- Group names have dots replaced with underscores (e.g., `example.io` → `example_io`)

These files are auto-generated — do not edit them manually. They are regenerated every time you run `kapibuild api generate`.

### src/controllers/

Directory for controller implementations. Initially contains only `mod.rs`. Use `kapibuild controller generate` to create controller scaffolding here. Each generated controller file follows the pattern `<kind>_controller.rs`.

### schemas/

Directory for generated JSON schemas. Use `kapibuild api generate` to compile types into schemas.

### src/main.rs

Entry point with Manager setup:
```rust
use kapi_client::client::KapiClient;
use kapi_controller::manager::Manager;

mod api;
mod controllers;
mod types;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error + Send + Sync>> {
    tracing_subscriber::fmt::init();
    
    let client = KapiClient::new("http://localhost:8080")?;
    let mut manager = Manager::new(client);
    
    // TODO: Register controllers here
    // manager.controller_for(key).reconciler(my_reconciler).register();
    
    manager.start().await?;
    
    Ok(())
}
```

## Usage

```bash
# Initialize in current directory
kapibuild init my-controller

# Initialize at specific path
kapibuild init /path/to/my-controller
```

The generated Cargo.toml uses version strings matching the kapibuild tool version. These resolve from crates.io once published, but during local development you may need to add path overrides in the generated project's Cargo.toml to point at your local kapi workspace.
