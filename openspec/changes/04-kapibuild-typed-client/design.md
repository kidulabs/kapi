## Context

Users have wrapper structs generated by kapi-derive. They need type-safe CRUD operations. The raw kapi-client works with StoredObject (JSON values), but users want to work with typed structs.

## Goals / Non-Goals

**Goals:**
- Provide generic CRUD trait over kapi-client
- Work with wrapper structs generated by kapi-derive
- Handle serialization of spec/status fields
- Handle deserialization of StoredObject back to typed structs
- Provide create, get, update, delete, list operations

**Non-Goals:**
- Controller scaffolding (separate command)
- Watch/streaming operations (future work)
- Schema registration

## Decisions

### 1. Trait design

**Decision:** Generic `TypedClient<T>` that wraps KapiClient and provides typed operations.

**Alternatives considered:**
- Extension trait on KapiClient (too magical, hard to test)
- Separate client per resource (too much code generation)

**Rationale:** Generic client is reusable, testable, and clear. Type parameter T is the wrapper struct.

### 2. Serialization strategy

**Decision:** Use serde for serialization. Spec/status fields serialized as JSON values within StoredObject.

**Rationale:** Matches kapi server's StoredObject format. Leverages existing serde integration.

### 3. Error handling

**Decision:** Return Result<TypedError> for all operations. TypedError wraps ClientError.

**Rationale:** Clear error types. Users can handle specific errors.

## Risks / Trade-offs

**[Risk] Generic client may be too abstract** → Mitigation: Start simple. Add helpers as needed.

**[Trade-off] Generic vs specific** → Generic is more flexible but may be harder to use. Can add resource-specific helpers later.
